<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| scripts/setup-storage.ts → Supabase Storage API | Uses service role key — bypasses RLS; must run server-side only |
| enable-rls.sql → PostgreSQL | Executed via Prisma db execute or Supabase SQL Editor — DML with service role |
| RLS enabled (deny-by-default) → all queries | No policies yet — ALL queries will be denied until Phase 2 adds auth policies. This is the correct foundation behavior. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Elevation of Privilege | Storage buckets publicly accessible | mitigate | Both buckets created with public: false; signed URLs required to access assets; public access policy explicitly disabled |
| T-03-02 | Elevation of Privilege | Tables accessible without RLS | mitigate | enable-rls.sql must be run before Phase 2 adds any auth logic; verified by pg_class.relrowsecurity = true query |
| T-03-03 | Information Disclosure | RLS enabled but no policies yet | accept | Deny-by-default is the CORRECT foundation state; no policies means no data can be read or written through the API until policies are added in Phase 2 — this is intentional and safe |
| T-03-04 | Tampering | scripts/setup-storage.ts using service role | mitigate | Script is in scripts/ directory (not in app/ or api/); service-role client created via lib/supabase/service.ts which has "server-only" guard; script never deployed to Vercel |
| T-03-SC | Tampering | npm install tsx | mitigate | tsx is a well-known TypeScript executor (esbuild-based); verify on npmjs.com — official package with >10M weekly downloads before install |
</threat_model>
